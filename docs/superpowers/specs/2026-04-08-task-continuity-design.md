# Design: Task Continuity — Storage, Memory & Git Operations
Date: 2026-04-08

## Overview

Three problems solved together:

1. **`.lacc` path resolution** — a consistent place for repo-scoped LACC config, skills, workflows, and task artifacts. Lives inside the repo (local mode) or in a global store namespaced by a user-defined name (global mode).
2. **Memory snapshots** — a concise `memory.md` per task, generated by Claude, that survives worktree deletion and provides context for future work.
3. **Explicit worktree retention** — no auto-deletion. Users manually choose what to keep when archiving a task.

---

## Section 1 — Storage Model & Path Resolution

### Two modes, one interface

Every repo registered in LACC has a `.lacc` root. Once resolved, the rest of the system treats both modes identically.

**Resolution order when a repo is opened or added:**
1. `<repo>/.lacc/` exists → **local mode** (committed to git)
2. `~/.lacc-data/repos/<name>/` exists in registry → **global mode**
3. Neither → prompt user to init (see Repo Init below)

`globalLaccPath` defaults to `~/.lacc-data/` and is configurable in global settings (e.g. pointing to a Dropbox or synced folder).

### Repo naming (global mode)

Repos in global mode are identified by a **user-defined flat name**, not an auto-derived namespace. When adding a repo, the name defaults to the folder name and is editable. If it conflicts with an existing name, the user is prompted to choose a different one.

A registry file maps repo paths to names:

```json
// ~/.lacc-data/registry.json
{
  "/Users/ivan/code/frontend": "acme-frontend",
  "/Users/ivan/code/my-app": "my-side-project"
}
```

Git remote URL is stored for informational context (shown in UI, passed to meta-Claude) but is never used as a storage key. This means local repos that later get a remote retain the same name without any migration.

### Directory structure (both modes, same layout)

```
<laccRoot>/
  config.yml                    ← repo-level LACC config
  workflows/                    ← repo workflow overrides
  tasks/
    <taskId>/
      .spec.md
      .plan.md
      .review.md
      memory.md
  .claude/
    commands/                   ← repo-specific step overrides
    skills/                     ← repo-specific skill overrides
```

For local mode: `<laccRoot>` = `<repo>/.lacc/`
For global mode: `<laccRoot>` = `~/.lacc-data/repos/<name>/`

### `config.yml` defaults

```yaml
commitLacc: true        # local mode only — commit .lacc/ to git
baseBranch: main
defaultWorkflow: null   # null = user picks at spawn time
```

### Repo init scaffolding

When user chooses to init, LACC creates:
- `config.yml` with defaults
- `workflows/`, `tasks/` directories
- `.claude/commands/`, `.claude/skills/` directories

No starter content is copied — starter skills live globally in `~/.lacc-data/` and are available to all repos via the container mount.

## Section 2 — Container Mounting & Agent Context

Every task agent gets three directory contexts: the worktree, the task's `.lacc/` folder, and the global LACC library.

### Container binds

**Local mode** (`.lacc/` is inside the worktree already):
```typescript
Binds: [
  `${worktreePath}:/workspace`,
  `${claudeDir}:/home/node/.claude`,
  `${claudeJson}:/home/node/.claude.json`,
  `${sshDir}:/home/node/.ssh:ro`,
  `${repoPath}:/original-repo:ro`,
  `${laccDataDir}:/lacc-global:ro`,          // global skills/workflows
]
```

**Global mode** (task storage is external):
```typescript
Binds: [
  `${worktreePath}:/workspace`,
  `${taskStoragePath}:/workspace/.lacc`,     // mounted inside workspace
  `${claudeDir}:/home/node/.claude`,
  `${claudeJson}:/home/node/.claude.json`,
  `${sshDir}:/home/node/.ssh:ro`,
  `${repoPath}:/original-repo:ro`,
  `${laccDataDir}:/lacc-global:ro`,          // global skills/workflows
]
```

From inside the container, both modes look identical:
```
/workspace/           ← repo code
/workspace/.lacc/     ← task storage (config, tasks/<id>/, workflows/)
/original-repo/       ← read-only original repo
/lacc-global/         ← user's global LACC library
```

### `--add-dir` flags passed to `claude -p`

```
--add-dir /workspace/.lacc      ← repo workflows, step overrides, task artifacts
--add-dir /lacc-global          ← global skills, global workflows, global steps
```

Combined with the agent's cwd `/workspace`, Claude Code's resolution order:
```
1. /workspace/.claude/          ← shared repo's own skills/commands (highest priority)
2. /workspace/.lacc/.claude/    ← repo LACC step overrides
3. /lacc-global/.claude/        ← user's global skills and steps
```

This means shared repos with existing `.claude/` skills and commands (including those that use MCPs like GitLab or Jira) work without any changes. The global LACC skills layer on top additively.

### Template variables available to agents

```
{{task_dir}}   → /workspace/.lacc/tasks/<taskId>/
{{spec}}       → /workspace/.lacc/tasks/<taskId>/.spec.md
{{plan}}       → /workspace/.lacc/tasks/<taskId>/.plan.md
{{review}}     → /workspace/.lacc/tasks/<taskId>/.review.md
{{memory}}     → /workspace/.lacc/tasks/<taskId>/memory.md
```

Substituted in step prompts before the agent receives them.

## Section 3 — Worktree Retention & Archive

### No auto-deletion

Worktrees are never automatically flagged for deletion. All retention management is manual via the UI. The existing `worktreeAutoDeleteHours` config is removed.

Containers are still cleaned up automatically when a task reaches a terminal state (they are expensive). Only the worktree (cheap disk space) is retained until the user acts.

### Four retention levels

When a user archives a task they choose a retention level:

| Level | Worktree | `.spec`, `.plan`, `.review` | `memory.md` |
|---|---|---|---|
| `alive` | ✅ | ✅ | ✅ |
| `archived` | ❌ | ✅ | ✅ |
| `summary` | ❌ | ❌ | ✅ |
| `deleted` | ❌ | ❌ | ❌ |

**Use cases:**
- `alive` — task is in QA or review, branch and full context still needed
- `archived` — work merged, worktree no longer needed, artifacts kept for reference
- `summary` — lightweight business/decision record, only the memory snapshot kept
- `deleted` — clean everything

### API

```
POST /tasks/:id/archive
Body: { level: 'archived' | 'summary' | 'deleted' }
```

Does not affect `tasks.status`. Git operations are completely independent of archive state.

### Git operations

Available whenever a worktree exists, regardless of task status. Never touch task status.

```
POST /tasks/:id/git/rebase      Body: { branch?: string }
POST /tasks/:id/git/pull
POST /tasks/:id/git/push
POST /tasks/:id/git/reset       Body: { hard?: boolean }
POST /tasks/:id/git/stash
POST /tasks/:id/git/stash/pop
```

Conflict response:
```json
{
  "ok": false,
  "conflict": true,
  "conflictedFiles": ["src/auth/jwt.ts"],
  "message": "Rebase conflict in 1 file. Resolve manually or reset."
}
```

## Section 4 — Memory Snapshots

### What it is

A `claude -p` call (haiku-class, ~$0.02-0.05) that distils the current task state into a concise `memory.md`. Available at any point: mid-task, before a rebase, or on completion.

### What it produces

```markdown
# feat/auth-jwt — Memory Snapshot
Generated: 2026-04-08 14:32

## What was built
JWT refresh token flow. Added /auth/refresh endpoint, token rotation
strategy, and middleware guards on protected routes.

## Key decisions
- 7-day refresh token expiry (agreed with user, original spec said 1h)
- httpOnly cookies over localStorage for security
- Refresh tokens stored in Redis with user-id index for revocation

## Files changed
- src/auth/jwt.ts (+82/-12)
- src/auth/middleware.ts (+34)
- src/auth/refresh.ts (+67)
- tests/auth.test.ts (+145)

## Deferred / known issues
- ENG-445: Token cleanup on logout
- Mobile Safari cookie issue needs investigation

## Context for future work
Schema in db/migrations/004_refresh_tokens.sql.
Middleware expects Authorization header OR cookie, not both.
```

### Generation inputs

- Original task prompt
- `.spec.md`, `.plan.md`, `.review.md` (if they exist)
- `git diff` against base branch (file list + line counts)
- User messages from the task feed (decisions, feedback given)

### API

```
POST /tasks/:id/memory-snapshot   → generates and saves memory.md, returns content
GET  /tasks/:id/memory            → reads current memory.md, 404 if none
```

Multiple snapshots overwrite — only the latest is kept. For local mode repos, `memory.md` is committed to git alongside the other task artifacts.

### Archive reference in follow-up tasks

Step prompts can reference previous task memory:

```
{{archive:<taskId>}}    → inlines memory.md from that task
{{archive:latest}}      → inlines memory.md from most recent task on same repo
```

## Section 5 — UI Changes

### Action bar — three independent zones

```
Agent controls   [Pause] [Kill] [Resume] [Restart]
                 driven by task status

Git operations   [Rebase] [Pull] [Push] [Stash]
                 shown whenever worktree exists, regardless of status

Lifecycle        [Save memory] [Archive ▾]
                 shown for terminal state tasks
```

`Archive ▾` opens a dropdown with the four retention level options.

Git history per task (showing which git operations were run and when) is a future addition.

### Memory tab

New tab in `DetailPanel` when `memory.md` exists:

```
[feed]  [workflow]  [diff]  [memory]  [preview]
```

Renders `memory.md` via the existing markdown renderer. Read-only. "Regenerate" button re-runs the snapshot and refreshes the tab.

### Repo init modal

Shown when adding a repo with no existing `.lacc/` configuration:

```
Set up .lacc for this repo

○ Local   Create .lacc/ inside the repo (commit to git)
○ Global  Store under name: [acme-frontend]
          in ~/.lacc-data/repos/

[Cancel]  [Init]
```

Name field shown only for Global option. Name defaults to folder name; conflicts prompt for a different name.

### Archive modal

```
Archive task?

Branch feat/auth-jwt will not be deleted — push first if needed.

○ Archive        Keep memory + all artifacts, remove worktree
○ Summary only   Keep memory.md only, remove everything else
○ Delete all     Remove everything including memory

[Cancel]  [Confirm]
```

### Task list — archive state indicator

Small visual indicator per task showing retention state (`alive`, `archived`, `summary`). Allows users to see at a glance what is retained without opening the task. Uses existing status dot + label mechanic.

## Implementation Phases

### Phase 1 — Foundation (backend only)
- `.lacc` path resolution logic (local vs global)
- `registry.json` for repo → name mapping
- Repo config (`config.yml`) read/write
- Worktree retention: remove auto-flagging, add `POST /tasks/:id/archive`
- Container mounting: `taskStoragePath` bind + `/lacc-global` bind
- `--add-dir` flags wired into `claude -p` invocation
- Template variables (`{{task_dir}}`, `{{spec}}`, etc.)

### Phase 2 — Memory & Git ops
- Memory snapshot generation (`POST /tasks/:id/memory-snapshot`)
- Memory read endpoint (`GET /tasks/:id/memory`)
- Git operations routes (rebase, pull, push, reset, stash, stash/pop)
- `{{archive:}}` template variable resolution

### Phase 3 — UI
- Action bar restructure (three zones)
- Memory tab in `DetailPanel`
- Archive modal
- Repo init modal (triggered from settings/repo add flow)
- Task list archive state indicator

---

## Key Constraints

- Git operations run on host via `simple-git`, not inside the container
- Rebase conflicts must be resolved in an IDE — LACC surfaces them only
- Branch is never deleted on archive — user pushes first if needed
- MCP passthrough requires no special plumbing — `~/.claude/` mount + container binaries handle it
- `simple-git` added as a new dependency in the orchestrator package
